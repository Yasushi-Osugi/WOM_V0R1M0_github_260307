# WOM Planning Engine Architecture

## Overview

This document describes the architecture of the **WOM Planning Engine**.

The WOM Planning Engine is the computational core of WOM (Weekly Operation Model).
It integrates demand generation, supply chain flow simulation, plan evaluation,
and decision search into a single planning loop.

Unlike traditional planning systems, WOM is designed as a **closed-loop planning engine**.

The core cycle is:

Scenario → Demand → Flow → State → Evaluation → Resolver → Operator → Re-simulation

This loop allows WOM to function as both:

- a supply chain simulator
- a decision support engine
- a planning kernel for AI-assisted economic planning

# 2 Scenario Layer

The Scenario Layer defines the initial planning conditions.

Typical scenario parameters include:

- demand assumptions
- network topology
- production capacities
- pricing parameters
- policy constraints
- planning horizon

Scenario inputs define the **starting point of the planning simulation**.

---

# 3 Demand Model

Module:


demand_model.py


Responsibilities:

- generate demand signals
- apply price response functions
- simulate promotion or policy effects
- produce demand events

Example output:


DemandEvent
{
product_id
location
time
quantity
}


The demand model converts market assumptions into **structured demand events**.

---

# 4 Flow Engine

Module:


flow_engine.py


The Flow Engine is the **core simulation engine of WOM**.

Responsibilities:

- generate lots
- simulate production
- simulate shipment flows
- derive inventory levels
- compute capacity usage

Key design principle:


Flow / Event = source of truth
State = derived view


The engine processes **event streams** such as:


ProductionEvent
ShipmentEvent
ArrivalEvent
SaleEvent


These events generate the derived planning state.

---

# 5 State View

The State View represents the observable state of the system.

Examples:

- inventory levels
- backlog
- service level
- capacity utilization
- financial summaries

Important principle:

The state is **not directly edited**.

It is derived from flow events generated by the simulation.

---

# 6 Trust Event Detection

After the state is derived, the system detects potential planning issues.

Examples of trust events:


Stockout risk
Inventory overflow
Capacity overload
Supply delay
Demand spike


These events are stored as structured artifacts.

Example file:


trust_events.json


Trust events trigger planning interventions.

---

# 7 Evaluation Engine

Module:


evaluation.py


The Evaluation Engine scores the current plan.

Typical metrics:

- profit
- service level
- inventory stability
- risk exposure
- capacity balance

Example evaluation function:


U(plan) =
w1 * service_level

w2 * profit

w3 * inventory_cost

w4 * risk


The evaluation engine converts system states into **objective scores**.

---

# 8 Resolver

Module:


resolver.py


The Resolver is the **decision search engine**.

Responsibilities:

- interpret trust events
- generate candidate operators
- simulate alternative scenarios
- evaluate candidate plans
- select the best operator

Resolver workflow:


Detect problem
→ Generate candidate operators
→ Apply operator to simulation
→ Re-run flow engine
→ Evaluate result
→ Select best operator


This process transforms WOM from a simulation tool into a **planning engine**.

---

# 9 Operator Selection

Operators modify the flow of the system.

Examples:


increase production
shift shipment
reroute logistics
use buffer inventory
adjust price


Operators must follow a critical rule:


Operators modify flow events,
not the derived state.


This preserves simulation consistency.

---

# 10 Flow Update

Once an operator is selected, it modifies the event stream.

Example:


operator_spec.json


The system then re-runs the Flow Engine to produce a new state.

This creates the **planning feedback loop**.

---
